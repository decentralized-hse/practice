# GossipSync

Децентрализованный веб-хостинг на основе `rsync` и `SSH`.

Система позволяет публиковать контент (статические файлы), подписываться на
чужие ленты и становиться зеркалом для их распространения — без центрального
сервера, DNS или баз данных.

## Архитектура

Более подробное текстовое описание архитектуры и мотивации можно прочитать в файле `rsync_ssh_decentralized_web.md`.

Сеть состоит из трёх ролей (один узел может совмещать все три):

```
┌─────────────────┐       rsync + SSH       ┌─────────────────┐
│    Издатель      │ ◄───────────────────── │   Подписчик     │
│  ~/public_feed/  │                         │  ~/feeds/alice/ │
│  - post1.md      │                         │  - post1.md     │
│  - mirrors.txt   │                         │  - mirrors.txt  │
└─────────────────┘                         └────────┬────────┘
                                                     │
                                              становится зеркалом
                                                     │
                                                     ▼
                                            ┌─────────────────┐
                                            │     Зеркало     │
                                            │ Другие качают   │
                                            │ отсюда, а не от │
                                            │ оригинала       │
                                            └─────────────────┘
```

**Издатель** — создаёт контент в директории `public_feed/`. Настраивает
ограниченный SSH-доступ через `rrsync` (read-only rsync).

**Подписчик** — хранит список лент в текстовом файле, по cron запускает
`rsync` для каждой подписки. Скачанные ленты хранятся локально.

**Зеркало** — подписчик, который открывает SSH-доступ к своей копии ленты.
Адрес зеркала записывается в `mirrors.txt`. Другие подписчики случайным
образом выбирают зеркало вместо оригинала — это снижает нагрузку на автора.

## Требования

- **Компилятор**: GCC 7+ или Clang 5+ с поддержкой C++17
- **rsync** — для синхронизации файлов
- **ssh** — OpenSSH клиент (для подписчиков)
- **sshd** + **rrsync** — OpenSSH сервер (для издателей/зеркал)

## Сборка

```bash
make
```

Или вручную:

```bash
g++ -std=c++17 -Wall -Iinclude src/main.cpp -o gossipsync
```

## Структура проекта

```
gossipsync/
├── Makefile
├── README.md
├── src/
│   └── main.cpp           # точка входа, разбор команд
└── include/
    ├── common.hpp          # общие типы и утилиты
    ├── publisher.hpp       # инициализация и публикация
    ├── subscriber.hpp      # подписки и синхронизация
    └── mirror.hpp          # зеркалирование
```

Данные хранятся в `~/.gossipsync/`:

```
~/.gossipsync/
├── public_feed/            # ваша публичная лента
│   ├── index.html
│   ├── mirrors.txt         # список зеркал этой ленты
│   └── 2026-03-12-post.md
├── feeds/                  # скачанные ленты
│   ├── alice/
│   └── bob/
├── mirrors/                # конфиги активных зеркал
├── subscriptions.txt       # список подписок
├── guest_key               # SSH-ключ для гостевого доступа
└── setup-guest-ssh.sh      # скрипт настройки SSH
```

## Использование

### Издатель

```bash
# Инициализация: создаёт директории, SSH-ключ, скрипт настройки
./gossipsync init

# Публикация поста (тело вводится из stdin)
./gossipsync post "Мой первый пост"

# Просмотр содержимого ленты
./gossipsync list-feed

# Настройка гостевого SSH (от root)
sudo ~/.gossipsync/setup-guest-ssh.sh
```

### Подписчик

```bash
# Подписаться на ленту
./gossipsync subscribe alice guest_sync@alice.example.com:/~/public_feed/

# Посмотреть подписки
./gossipsync list-subs

# Синхронизировать все ленты
./gossipsync sync

# Синхронизировать со случайной задержкой (для cron)
./gossipsync sync --jitter

# Отписаться
./gossipsync unsubscribe alice

# Показать строку для crontab
./gossipsync setup-cron
```

### Зеркалирование

```bash
# Стать зеркалом скачанной ленты
./gossipsync mirror alice guest_sync@myserver.net:/~/alice_mirror/

# Мои активные зеркала
./gossipsync list-mirrors

# Известные зеркала конкретной ленты
./gossipsync feed-mirrors alice
```

## Тестирование с помощью Docker

Чтобы не настраивать `sshd` на хосте, можно поднять «узел‑издатель» в Docker‑контейнере и тестировать `gossipsync` как подписчика.

### 1. Подготовка

В корне проекта уже есть файлы:

- `Dockerfile` — образ Debian + `openssh-server` + `rsync`
- `docker-compose.yml` — сервис `author` с проброшенным портом `2222`
  (на хостах порт `22` часто уже занят системным `sshd`, поэтому для
  тестов удобнее использовать отдельный порт `2222`).

Соберите бинарник и инициализируйте локальный узел:

```bash
make
./gossipsync init
```

Поднимите контейнер‑автор:

```bash
docker compose up -d
```

Проверьте, что SSH внутри контейнера работает:

```bash
ssh -i ~/.gossipsync/guest_key -p 2222 root@localhost 'rsync --version'
```

### 2. Подписка на Docker‑автора и синхронизация

Очистите/создайте подписку `author` на контейнер (GossipSync внутри контейнера
инициализирует ленту по пути `~/.gossipsync/public_feed`, т.е.
`/root/.gossipsync/public_feed`):

```bash
./gossipsync unsubscribe author 2>/dev/null || true
./gossipsync subscribe author "root@localhost:/root/.gossipsync/public_feed/"
```

Синхронизация идёт через нестандартный SSH‑порт `2222`, который `gossipsync` читает из переменной окружения `GOSSIPSYNC_SSH_PORT`:

```bash
export GOSSIPSYNC_SSH_PORT=2222
./gossipsync sync --jitter
```

Ожидаемый результат — успешный `rsync` и появление файлов в локальной
копии:

```bash
ls ~/.gossipsync/feeds/author
```

### 3. Минимальный e2e‑сценарий

1. В контейнере (как `root`) положить файл в ленту:
  ```bash
   docker exec -it gossipsync-author bash
   cd /root/.gossipsync/public_feed
   echo "hello from container" > test.txt
   exit
  ```
2. На хосте обновить подписку:
  ```bash
   export GOSSIPSYNC_SSH_PORT=2222
   ./gossipsync sync --jitter
  ```
3. Убедиться, что файл пришёл:
  ```bash
   ls ~/.gossipsync/feeds/author
   cat ~/.gossipsync/feeds/author/test.txt
  ```

Такой сценарий проверяет, что `gossipsync` как подписчик корректно работает поверх `rsync+ssh` против реального `sshd`, но при этом не требует настраивать сервер на вашей машине.

## Как работает синхронизация

1. Команда `sync` читает `subscriptions.txt`
2. Для каждой подписки проверяет `mirrors.txt` в локальной копии ленты
3. С вероятностью 50% выбирает случайное зеркало вместо оригинала
4. Запускает `rsync -avz --delete` через SSH-туннель
5. Если зеркало недоступно — автоматически откатывается на оригинал

Флаг `--jitter` добавляет случайную задержку 0–5 минут перед синхронизацией.
Это предотвращает ситуацию «шквала запросов» (thundering herd), когда
сотни cron-задач одновременно обращаются к серверу.

## Настройка SSH-доступа (подробно)

Скрипт `setup-guest-ssh.sh`, создаваемый при `init`, делает следующее:

1. Создаёт системного пользователя `guest_sync`
2. Находит утилиту `rrsync` (restricted rsync)
3. Добавляет SSH-ключ в `authorized_keys` с ограничениями:
  - `command="rrsync -ro /path/to/feed"` — только чтение rsync
  - `no-agent-forwarding` — запрет проброса агента
  - `no-port-forwarding` — запрет проброса портов
  - `no-pty` — запрет терминала

Таким образом, подключившийся пользователь может **только** скачивать файлы
из указанной директории и ничего больше.

## Известные ограничения

| Проблема | Описание | Обходной путь |
|----------|----------|---------------|
| NAT | Зеркала должны быть доступны по SSH | IPv6, Tor, Tailscale |
| Шквал запросов | Одновременные cron-задачи | Флаг `--jitter` |
| Подмена контента | Зеркало может изменить файлы | GPG-подпись (не реализовано) |
| Безопасность SSH | Открытый SSH-порт — вектор атаки | rrsync + ForceCommand |
| Файлы-призраки | Удалённые файлы возвращаются через зеркала | Флаг `--delete` в rsync |
