# `BitTorrent` на оборот

## Recup устройства `BitTorrent`

`BitTorrent` устроен так:

- есть фиксированный контент,
- есть открытое и постоянно меняющееся множество пиров,
- цель - максимально эффективно распространить один и тот же объект.

Обратная постановка выглядит так:

- есть фиксированное множество доверенных участников,
- контент может быть любым и постоянно меняться,
- цель — построить отказоустойчивую, доступную, распределённую систему хранения и доставки данных внутри стабильной доверенной сети.

## Как бы выглядела такая система

### 1. Участники и домены отказа

Есть большая группа доверенных лиц, которые знают друг о друге и могут обмениваться ресурсами.

Все узлы разбиваются на небольшое число доменов отказа:

- по географии,
- по провайдеру,
- по типу площадки,
- по административной принадлежности,
- или по любому другому критерию, который коррелирует с совместными отказами.

После этого система будет уже зависеть не от отказов конкретных узлов, а от коррелированных отказов.

### 2. Базовая единица хранения — VDisk

Физические диски или их ёмкость дробятся на условные куски, например по 40 GB. Эти куски называем VDisk.

Это даёт несколько плюсов:

- унифицированное планирование размещения,
- простую балансировку,
- независимость от реальной структуры железа,
- удобную миграцию и ребалансировку.

По сути VDisk — это некий стандартный контейнер хранения, на который можно навешивать правила размещения, ремонта и вытеснения.

### 3. Группа хранения

Дальше 8 VDisk объединяются в группу хранения.

Внутри группы данные пишутся не просто копированием, а через стирающий код (erasure coding). Например:

- схема `4+2` — на каждые 4 блока данных добавляем 2 блока избыточности;
- или схема `6+2`, если хочется использовать все 8 VDisk симметрично;
- в общем случае важно не столько точное число, сколько свойство:  
  группа переживает потерю до 2 VDisk без потери данных.

Идея: вместо полной репликации мы получаем:

- лучшую эффективность по месту,
- предсказуемую отказоустойчивость,
- возможность фонового восстановления.

### 4. Размещение по доменам отказа

Ключевая часть дизайна — одна группа хранения должна включать по одному VDisk из разных доменов отказа.

То есть группа формируется так, чтобы её элементы были максимально разведены:

- по географии,
- по сети,
- по владельцам,
- по площадкам.

Тогда отказ одного домена не убивает сразу целую группу.

Если домен пропадает:

- часть VDisk становится недоступной,
- но данные восстанавливаются из оставшихся фрагментов,
- система начинает автоматический repair,
- и недостающие фрагменты реплицируются или пересобираются в другие домены.

Это позволяет обеспечить надёжное хранение данных при условии отказов в пределах одного / двух доменов отказа.

### 5. Модель данных: immutable chunks + mutable metadata

Тут получается самая большая разница с `BitTorrent`, в котором неизменяемый контент: хэш однозначно определяет объект.

Для хранения обычных файлов, папок, версий их их обновления используется двухслойный подход:

#### Нижний слой

- файл режется на чанки,
- чанки адресуются по хэшу,
- данные внутри считаются неизменяемыми.

#### Верхний слой

- есть манифест файла,
- есть версия,
- есть ссылка вида "текущая активная версия файла = такая-то",
- сама эта ссылка уже изменяемая, но подписанная и версионируемая.

### 6. Загрузка и публикация новой версии

#### Процесс записи

1. Файл версионируется.
2. Новая версия режется на чанки.
3. Чанки раскладываются по группам хранения.
4. Система ждёт подтверждения от кворума.
5. Только после этого продвигается указатель на активную версию.

То есть запись работает по модели:

- сначала данные становятся надёжно размещёнными,
- потом метаданные делают версию видимой.

Такой паттерн исключает ситуацию, когда новая версия сохранилась частично, а старая уже недоступна.

### 7. Параллельные изменения

Параллельные модификации будут завершаться через оптимистичную блокировку.

Примерно так:

- клиент читает текущую версию `v17`,
- готовит `v18`,
- при коммите говорит: записать `v18`, если текущая активная версия всё ещё `v17`,
- если за это время кто-то уже успел продвинуть `v18` или `v19`, коммит отклоняется.

## Применимы ли rarest-first и tit-for-tat?

### Rarest-first

В некотором другом смысле - да.

В BitTorrent rarest-first работает, потому что много людей одновременно хотят один и тот же объект. Нужно сначала раздавать самые редкие куски, чтобы рой не деградировал.

В обратной системе получилась другая проблема: как поддерживать надёжность и доступность большого множества разных объектов.

Поэтому rarest-first переопределяется так:

- Сначала восстанавливать и распространять те чанки или те фрагменты, у которых меньше всего доступных копий / кодовых фрагментов.

То есть он становится не алгоритмом загрузки, а алгоритмом:

- ремонта,
- ребалансировки,
- поддержания долговечности данных.

#### Где он полезен в системе

1. При восстановлении после отказа домена: сначала чинить самые редкие или самые уязвимые фрагменты.
2. При фоновом repair: поднимать целевой уровень избыточности у наиболее недореплицированных объектов.
3. При скачивании крупных файлов из нескольких доменов: можно всё ещё выбирать редкие фрагменты первыми, чтобы не упираться в один источник.

Таким образом:  
rarest-first применим, но как политика устойчивости, а не как основной рыночный механизм роя.

### Tit-for-tat

В классическом bittorrent-формате - нет.

Классический tit-for-tat означает:

- "ты даёшь мне сейчас — я даю тебе сейчас",
- немедленное взаимное вознаграждение,
- борьба с фрирайдерами на уровне текущего обмена.

В нашей системе это слишком узко, потому что взаимодействие:

- долгосрочное,
- асимметричное,
- распределённое по времени,
- и часто косвенное.

Например:

- сегодня я храню твои кодовые фрагменты,
- завтра ты кешируешь мой популярный файл,
- через неделю кто-то третий использует твой узел как источник для восстановления моих данных.

Здесь больше подходит не tit-for-tat, а:

- квоты,
- репутация (например как часто этот узел недоступен).

#### На практике можно сделать так

- у каждого участника есть базовая квота,
- вклад учитывается по:
  - объёму хранения,
  - аптайму,
  - выданному трафику,
  - участию в repair,
  - полезности кеша,
  - стабильности узла,
- за больший вклад даются:
  - более высокие лимиты,
  - более высокий приоритет ремонта,
  - более быстрые выдачи,
  - больший объём допустимого хранения данных.

То есть в отличие от BitTorrent система должна понимать в долгосрочной перспективе на каком узле лучше хранить данные.

## Что сломается первым?

Если пытаться просто перевернуть BitTorrent, сломаются несколько его базовых допущений.

### 1. Сломается swarm effect

BitTorrent силён тем, что:

- много людей хотят одно и то же,
- у всех частично совпадает интерес к объекту,
- можно выгодно обменивать куски.

В нашей системе:

- контент гораздо более разнородный,
- многие объекты нужны одному владельцу и ещё, может быть, нескольким подписчикам,
- массового «роя» вокруг одного файла обычно нет.

Поэтому основной выигрыш BitTorrent — огромный параллелизм на одном объекте — исчезает.

У нас на первый план выходят не swarm-механики, а:

- правильное размещение,
- кодирование,
- ремонт,
- кеширование,
- домены отказа,
- и планирование сетевых путей.

### 2. Сломается простая идентичность объекта

В BitTorrent объект фиксирован: у него есть hash, и всё просто.

В нашей системе файлы живые:

- они версионируются,
- обновляются,
- переименовываются,
- могут редактироваться параллельно.

Значит, просто хеша для идентификации обхекта уже недостаточно.

Приходится вводить:

- манифесты,
- указатели на текущую версию,
- политику публикации,
- optimistic locking,
- возможно журнал изменений.

Именно слой mutable metadata становится сложнее, чем сам перенос байтов.

### 3. Сложнее станет discovery и placement

В BitTorrent вопрос простой: есть torrent/metainfo — дальше ищем пиров.

У нас вопросы сложнее:

- где физически лежат фрагменты?
- в каких доменах отказа?
- хватает ли избыточности?
- где сейчас лучший источник для скачивания?
- когда нужно repair?
- какие объекты кэшировать?
- что можно вытеснить из follower-cache?
- какая группа перегружена?

Таким образом задача смещается из "быстро качать" в "правильно управлять состоянием кластера".

### 4. Метаданные становятся почти важнее самих данных

Даже если контент зашифрован, система всё равно раскрывает многое:

- кто с кем связан,
- какие узлы что хранят,
- что часто скачивают,
- какие объекты становятся активными,
- где происходят ремонт и репликация.

Это может быть серьёзным вопросом безопасности, но для доверенной сети выглядит не критично.

### 5. Availability может проседать без активного repair

Если не делать фоновый ремонт, то даже хорошая схема кодирования начнёт деградировать:

- домены могут временно исчезать,
- узлы могут неделями быть оффлайн,
- follower-cache может вытесняться,
- географически удалённые источники могут стать узким местом.

По этому важно контролировать и поддерживать устойчивость такой системы.

## Как сделать такую систему рабочей

### 1. Разделить data plane и control plane

#### Data plane

Отвечает за:

- чанки,
- кодовые фрагменты,
- загрузку,
- скачивание,
- кеширование,
- repair.

#### Control plane

Отвечает за:

- метаданные,
- версии,
- манифесты,
- размещение,
- учёт доменов отказа,
- кворумы,
- продвижение активной версии,
- учёт вкладов участников.

Это критично: иначе логика версии файла и логика размещения кодовых фрагментов смешаются и быстро превратятся в хаос.

### 2. Ввести явный placement policy

Для каждой группы хранения система должна учитывать:

- домен отказа,
- свободное место,
- текущую нагрузку,
- пропускную способность,
- надёжность узла,
- latency до целевых клиентов,
- историческую стабильность.

Тогда выбор VDisk становится не случайным, а управляемым.

Пример цели:

- в одну группу не попадают два VDisk из одного домена отказа;
- по возможности не попадают узлы одного владельца;
- желательно обеспечивается сетевое разнообразие;
- горячие данные можно размещать ближе к потребителям.

### 3. Repair должен быть фоновым и приоритетным

При отказе домена или VDisk система должна не просто «ждать, пока всё само вернётся», а запускать:

- детект деградации,
- оценку риска,
- пересборку недостающих фрагментов,
- размещение в новых доменах,
- контроль выхода на целевой уровень устойчивости.

Приоритеты ремонта можно задавать по формуле из:

- числа оставшихся фрагментов,
- важности объекта,
- популярности,
- времени с последней полной верификации,
- вероятности повторного отказа.

Тут очень уместен rarest-first очень уместен: чинить сначала то, что ближе всего к потере.

### 4. Кворум записи и commit активной версии

#### Запись

1. Клиент отправляет чанки и кодовые фрагменты.
2. Узлы подтверждают durable-write.
3. Когда достигнут write quorum, создаётся новый манифест версии.
4. Активная версия переключается атомарно.

#### Чтение

- чтение по возможности идёт с ближайших или самых дешёвых источников,
- но манифест должен браться из согласованного источника метаданных.

Это защищает от подвисших версий и частично записанных файлов.

### 5. Версионирование и optimistic locking

Файлы можно представить так:

- `file_id`
- `version`
- `manifest_hash`
- `parent_version`
- `state = staging | committed | active | obsolete`

Тогда обновление выглядит аккуратно:

- новая версия загружается как `staging`,
- после кворума становится `committed`,
- затем CAS-операцией продвигается в `active`,
- если `parent_version` уже не совпадает — конфликт, запись отклоняется.

### 6. Ускорение чтения: перекрытие и follower-cache

Здесь есть два уровня:

#### Базовый слой

- есть кодовые фрагменты, обеспечивающие сохранность.

#### Ускоряющий слой

- есть дополнительное перекрытие/кеширование,
- чтобы читать параллельно из нескольких доменов,
- снижать задержки,
- утилизировать разные каналы.

Это особенно полезно при геораспределённости.

#### Follower-cache

После скачивания файл или его чанки временно держатся в LRU-кеше у тех, кто недавно его читал.

Это фактически превращает потребителей в временных сидов, но уже не по bittorrent-логике, а по cache-логике.

Плюсы:

- горячие файлы читаются быстрее,
- снижается нагрузка на исходные группы хранения,
- можно обслуживать всплески чтения,
- часть repair можно делать быстрее, если рядом уже есть кэшированные данные.

В идеале правильное кеширование может обеспечить аналог CDN в социальной сети.

### 7. Нужен учёт ресурсов и справедливости

Даже в доверенной группе нельзя бесконечно полагаться только на добрую волю.

Нужно хотя бы мягко учитывать:

- сколько места участник предоставляет,
- сколько трафика отдаёт,
- насколько стабилен,
- как часто участвует в repair,
- сколько горячих данных кэширует.

И использовать это для:

- назначения квот,
- выбора приоритетов,
- балансировки нагрузки,
- планирования размещения.

## Во что в итоге превращается система

Если собрать всё вместе, получается примерно такая конструкция:

### Слой хранения

- VDisk как базовая единица,
- группы хранения из 8 VDisk,
- erasure coding,
- размещение по разным доменам отказа.

### Слой отказоустойчивости

- домены отказа,
- автоматический repair,
- перебалансировка при исчезновении домена,
- контроль минимального уровня избыточности.

### Слой данных

- файлы режутся на чанки,
- чанки неизменяемы,
- версии файлов описываются манифестами,
- активная версия переключается только после кворума.

### Слой конкурентного доступа

- optimistic locking,
- CAS на продвижение активной версии,
- конфликты записи завершаются ошибкой.

### Слой производительности

- параллельное скачивание из разных доменов,
- дополнительное перекрытие для более полной утилизации сети,
- follower-cache,
- LRU-кеширование горячих файлов.

### Слой кооперации

- долговременный учёт вклада,
- квоты,
- приоритеты,
- мягкая экономика вместо классического tit-for-tat.

## Короткий вывод по вопросам

### Как выглядела бы такая система?

Как доверенное геораспределённое хранилище, где данные режутся на чанки, кодируются стирающим кодом, раскладываются по VDisk в разных доменах отказа, версионируются и публикуются только после подтверждения кворума.

### Применим ли rarest-first?

Да, но не как в BitTorrent. Здесь он полезен прежде всего как политика:

- ремонта,
- фонового восстановления,
- поддержания достаточной избыточности,
- а иногда - как стратегия многопоточного скачивания.

### Применим ли tit-for-tat?

В классическом виде - нет. Его нужно заменить на:

- квоты,
- долгосрочный учёт вклада,
- репутацию,
- или социальное доверие.

### Что в системе сломается первым?

Не доставка чанков, а:

- отсутствие swarm-эффекта,
- сложность mutable metadata,
- размещение по доменам отказа,
- repair после отказов,
- и управление доступностью.

### Как сделать систему рабочей?

Через сочетание:

- immutable chunks,
- mutable signed manifests,
- quorum commit,
- optimistic locking,
- erasure coding,
- failure domains,
- background repair,
- follower-cache,
- и долговременного resource accounting.
