# Сравнение blob diffing в разных VCS

## Что тестировалось

Взял три системы контроля версий и посмотрел, как они работают с diff'ами файлов:

**Git** - всем известный, используется везде. Он работает на packfiles, дельты считает при `git gc`. Что интересно - он может искать похожий контент где угодно, а не только в одном файле.

**Mercurial (hg)** - тоже популярная система, менее популярная чем Git, ну и чуть проще устроена. Использует revlog формат - первая версия файла полная, остальные хранятся как дельты: для каждого файла свой `.i` файл с историей.

**SVN** - очень старая централизованная система, она все еще использует построчные дельты, да и каждая ревизия - отдельный файл в репе. Честно говоря уже устарела, но взять и потрогать для сравнения было прикольно.

## Эксперимент

Сделал простой тестовый файл `sample.txt` и создал 3 версии:
- v1: 10 строк (~250 байт)
- v2: поменял 2 строки, добавил 1 новую (11 строк, ~280 байт)
- v3: еще 2 изменения + 2 новых строки (13 строк, ~360 байт)

Закоммитил все это в каждую систему и посмотрел что получилось.

### Git

Команды для теста:
```bash
git init
# 3 коммита с разными версиями файла
git log --oneline
git diff HEAD~2 HEAD sample.txt
du -sh .git
git gc  # компрессия
git count-objects -v
```

Что получилось:
- До gc: 132 KB
- После gc: 116 KB (экономия ~12%)  
- Packfile всего 2 KB - это вообще круто, очень эффективно сжал
- 13 объектов запаковал

Diff между первой и третьей версией:
```
-This is the first version of the file.
+This is the THIRD version of the file.
-Line 3: some content here
-Line 4: more content
+Line 3: MODIFIED content here
+Line 4: COMPLETELY REWRITTEN
...
+Line 11: NEW LINE added
+Line 12: Another new line
```

Интересный момент - в packfile версии 1 и 2 сохранились как дельты к версии 3. То есть Git получается самостоятельно выбрал базу и оптимизировал все.

### Mercurial

```bash
export HGUSER="Test User <test@example.com>"  # hg требует юзера
hg init
# 3 коммита
hg log
hg diff -r 0 -r tip sample.txt
du -sh .hg
```

Результаты:
- Размер репы: 88 KB - самый маленький из всех!
- `sample.txt.i` всего 710 байт (все версии + дельты)
- Общий размер данных ~3 KB

Diff выглядит в целом так же как в Git (ибо unified format). История коммитов же выглядит более просте чем в гите, хеши короче

### SVN

```bash
svnadmin create repo
svn checkout file://$(pwd)/repo working-copy
cd working-copy
# 3 коммита
svn log
svn diff -r 1:3 sample.txt
du -sh ../repo
```

Результаты:
- Репозиторий: 148 KB - самый большой
- Рабочая копия: 152 KB (тоже хранит копии)
- Структура FSFS: каждая ревизия отдельный файл в `db/revs/0/`

История с ревизиями r1, r2, r3 - проще читать чем хеши, но занимает больше места. Diff выглядит почти так же как в Git, только вместо "diff --git" пишет "Index: filename". По сути те же + и - для изменений, в общем одно и тоже

## Сравнение

| Что смотрел | Git | Mercurial | SVN |
|-------------|-----|-----------|-----|
| Размер репы | 116 KB (после gc) | 88 KB ✔️ | 148 KB |
| Сжатие данных | ~2 KB (packfile) ✔️ | ~3 KB (revlog) | отдельные файлы |
| Скорость | быстро ✔️ | быстро ✔️ | медленно (нужна сеть) |
| Когда считает дельты | при gc/push | сразу при коммите ✔️ | при коммите |

### Что понравилось/не понравилось

**Git:**
- ➕ Самое эффективное сжатие, packfiles реально крутые
- ➕ Может искать похожие данные где угодно, не только в одном файле
- ➕ Все используют, куча инструментов
- ➖ Сложно устроен внутри, долго пришлось разбираться
- ➖ Нужно запускать gc руками для оптимизации

**Mercurial:**
- ➕ Самый компактный! 88 KB против 116 KB у git
- ➕ Проще понять как работает (revlog достаточно очевидный)
- ➕ Тоже быстрый
- ➖ Дельты только внутри одного файла
- ➖ Менее популярен и меньше тулзов под него

**SVN:**
- ➕ Номера ревизий проще чем хеши (r1, r2, r3)
- ➕ Понятная структура репозитория
- ➖ Самый большой размер (148 KB)
- ➖ Для многих операций нужен доступ к серверу
- ➖ Морально и технологически устарел, никто не использует для новых проектов (и слава богу :))

## Выводы

**По размерам:** Mercurial выиграл по общему размеру репы (88 KB), но Git лучше сжимает сами данные (packfile 2 KB vs revlog 3 KB). Ну, аSVN просто проиграл (148 KB) :P.

**По алгоритмам:** Git самый умный - может создавать дельты между любыми похожими объектами, даже из разных файлов. Mercurial и SVN только построчные дельты внутри одного файла, что проще но менее гибко.

**Что бы я выбрал:**
- Для реального проекта - **Git**, потому что все его используют и инструментов много
- Если нужна простота - **Mercurial**, имхо понять сильно проще, да и сам он компактнее
- **SVN** только если заставят на работе, для новых проектов не имеет смысла

**Скорость:** На маленьких данных все работают быстро. На больших репах Git и Mercurial будут намного быстрее SVN, ибо все локально и не нужно к серверу обращаться каждый раз
